## TL;DR

- 半導体のデジタル設計を本業とする新卒2年目が、2025年12月〜2026年2月の2ヶ月で **AZ-900 / AZ-104 / LinuC レベル1 / AWS SAA** の4資格を取得した学習記録です。
- クラウド実務経験ゼロからのスタート。通勤時間の動画視聴と業務後1時間を中心に、平日は合計3時間ペースで積み上げました。
- 「ハードウェア側で仕事をしている人間がクラウドを学ぶ意味はあるのか」という問いに、2ヶ月後の自分なりの答えを置いておきます。
- 同じようなバックグラウンドでクラウドやLinuxに踏み出したい方の参考になれば嬉しいです。

## はじめに

想定読者は以下を念頭に書きました。

- HW/組み込み・制御系出身で、クラウドに興味はあるが最初の一歩を踏み出せていない方
- 新卒〜若手エンジニアで、短期間に複数資格を計画している方
- Azure・AWS・Linuxの複合取得を検討している方

本記事で扱うのは、4資格の受験日・学習時間・使用教材・詰まった点、そしてHW寄りの立場でクラウドに触れて感じた世界観の違いです。技術解説よりも学習ログ寄りですが、公式教材だけでは埋めきれない「実際どうだったか」の空気感を共有できればと思います。

## なぜ2ヶ月で4資格を取ったのか

きっかけは次のような優先順でした。

1. **知的好奇心が先行**。業務で触れている半導体デジタル設計の外側、とくに上位レイヤー(OS/ネットワーク/クラウド)にどんな概念があるのかを体系的に知りたかった
2. **転職戦略の一部**。HW設計 × クラウド/Linux という組み合わせを、定量的に示せる資格として手元に揃えておきたかった
3. **単純に資格取得が面白かった**。短期で1つ取ると次が欲しくなる、というループが回った

並行進行を選んだ理由は、Azure・AWS・Linuxで重なる概念が多く、同時期に学ぶことで理解が強化されると考えたためです。実際、AZ-104の学習がAWS SAAの学習時間を大きく短縮してくれました。

無理に資格を取りに行ったというよりは、**興味を追いかけていたら結果的に4本取れていた**、という方が実感には近いです。

- **休日**: 移動中の1時間を中心にスポットで積む

業務は基本的に定時退社ペースで両立できていました。帰宅後にまとまった時間を確保していたわけではなく、通勤の移動と平日夜の限られた時間で積み上げています。

## 各資格のリアル

### AZ-900 (Azure Fundamentals)

| 項目 | 内容 |
| --- | --- |
| 受験日 | 2025/12/21 |
| 学習時間 | 約15時間(1週間) |
| 教材 | Microsoft Learn、Udemy(動画講座 + 模擬試験) |
| 難易度感 | 妥当 |

クラウドの概念学習そのものがほぼゼロからのスタートでした。SaaS/PaaS/IaaSの境界、リージョン、Availability Zoneなど、今振り返れば基礎の基礎と位置づけられる用語が、当時はほとんどすべて初耳でした。AZ-900の難易度自体は高くないものの、学習初期の負荷は決して軽くなかった、というのが正直な実感です。

HW寄りの視点でいちばん印象に残ったのは、**手軽にサーバーを立てて冗長化まで済ませられる**という体験でした。物理実装や筐体設計を一切介さず、クリック操作とコードだけで可用性が構成できるのは、普段デジタル回路のタイミング制約に向き合っている日常からすると、まるで異世界のスピード感でした。

### AZ-104 (Azure Administrator Associate)

| 項目 | 内容 |
| --- | --- |
| 受験日 | 2026/1/17 |
| 学習時間 | 約40時間 |
| 教材 | Microsoft Learn、Udemy(動画講座 + 模擬試験) |
| 難易度感 | 難 |

4資格のなかでもっとも負荷を感じた試験です。サービス数が多く、各サービス間の関係性を整理するだけで相当な時間を要します。加えてAZ-104は、画面の操作を問う出題形式が含まれており、**テキスト教材だけでは到達しづらい**のが特徴です。GUIの構成や項目のネストが頭に入っていないと、正解を選べない問題が混ざってきます。ここで助けられたのが、Udemyの動画講座で**ハンズオンの操作手順をそのまま見せてくれる**コンテンツでした。自前で環境を用意しきれない時期に、画面の流れや選択肢の位置感覚を掴むうえで大きく役立ちました。

試験数日前の段階で、すべてのサービスを完全に理解しきれないことを受け入れ、学習スタイルを **「理解中心」から「試験対策中心」に切り替え**ました。模擬試験を繰り返して出題パターンに体を慣らし、頻出サービスの操作手順を重点的に追い直す方針です。資格勉強としては本来望ましい姿ではないかもしれませんが、期限が迫った場面での有効な割り切りだったと思います。

### LinuC レベル1 (101 + 102)

| 項目 | 内容 |
| --- | --- |
| 受験日 | 101: 2026/1/11、102: 2026/1/24 |
| 学習時間 | 101: 約10時間 / 102: 約10時間(合計 約20時間) |
| 教材 | Linux教科書 LinuCレベル1 Version 10.0対応(あずき本)、Ping-t |
| 難易度感 | 易 |

業務で日常的にBashやLinuxサーバーに触れていたため、操作系コマンドの半分程度は既知でした。そのぶん、試験全体の負荷は4資格の中でもっとも軽く済んでいます。実際のところ、AZ-104の学習と並行しているあいだに自然と消化できた、という感覚に近いです。

ただし、業務では意識せずに済ませていた領域を体系的に問われる場面で、ギャップを実感しました。特に**ルーティング**や**インタフェース設定**といったネットワーク領域は、普段の業務では抽象化されて見えなくなっている部分が多く、仕組みを言語化して覚え直す必要がありました。あとは `systemctl` や `journalctl` などのシステム系、`ip route` のようなネットワーク系のコマンドも試験範囲に含まれており、業務では触れてこなかった領域を網羅的に押さえる必要もありました。

教材の使い方は当初、「あずき本を読み進めて、各単元が終わったらPing-tで演習」という進め方でした。ただ、途中から**単元の概要だけ掴んでPing-tの演習に飛び込む**方針に切り替えたところ、学習ペースが大きく上がりました。LinuCに限らず、**学習中盤で教材の使い方を見直す**のは大事だと思います。

### AWS SAA (Solutions Architect Associate)

| 項目 | 内容 |
| --- | --- |
| 受験日 | 2026/2/14 |
| 学習時間 | 約20時間 |
| 教材 | CloudTech(動画教材)、Ping-t |
| 難易度感 | 妥当 |

AZ-104の知識資産が大きく効いた1本でした。クラウドの基本サービスや設計原則はAzureとAWSで共通概念が多く、サービス名の対応を覚え直すだけで理解できる領域が少なくありません。結果として、学習時間はAZ-104の半分で収まっています。

教材選定ではUdemyを使いませんでした。Udemyの講座は基礎解説から始まるものが多く、AZ-104履修済みのレベル感に合うコースが見つけづらかったためです。**CloudTech**は解説の粒度が自分の前提に合っており、評判も良かったため採用しました。

苦戦した点は2つあります。ひとつは**出題形式の違い**です。AZ-104が画面操作中心なのに対し、SAAは長文のシナリオ問題が多く、読み解きのリズムが明確に異なります。もうひとつは**ベストプラクティスの把握**で、Well-Architected Frameworkの柱のうちコスト最適化の点数が特に低く出ました。ベストプラクティスは試験対策だけでは身につかない領域だと感じています。

SAAは後半でモチベーションが下がった試験でもあります。座学そのものの興味が続かなくなった時期があり、そのときに役立ったのが、**AWSの最新サービスに関する記事を試験範囲外でも読む**という回し方でした。とくにAmazon BedrockのRAG構築事例やエージェント構築、AWS IoT Coreの活用記事は、単純に読んでいて面白く、学習再開のきっかけになってくれました。直接的な試験対策にはなりませんが、「クラウドで何ができるのか」の解像度が上がることで、試験範囲の各サービスの意味合いも腹落ちしやすくなる効果があったように思います。

## 4資格連投のコツ

### スケジュール設計: 先に試験日を確定する

試験日を先に予約してしまうことで、学習ペースを強制的に試験日に合わせられます。これが一番効きました。資格学習は「いつか受けよう」だと永遠に着手できないので、試験日確定を最初のコミットメントに置くのが近道だと思います。試験間の間隔は、今回のペースであれば10〜14日程度が現実的な感覚でした。

### 教材遷移の判断: 自分のレベルに合わせて切り替える勇気

AZ-900とAZ-104はUdemyとMicrosoft Learn、LinuCはあずき本+Ping-t、SAAはCloudTech+Ping-t、と**試験ごとに教材を変えました**。これは意図的な選択で、前の試験で得たレベル感に対して、同じブランドの入門講座を使い続けると冗長になるためです。SAAでUdemyを選ばなかったのも同じ理由でした。教材を切り替える判断は、学習効率に直結する部分だと振り返って感じます。

### モチベーション管理: 興味範囲を広げて失速に備える

モチベ失速は後半ほど訪れやすいです。SAAのときに効いたのは、**試験範囲外のサービスや最新記事を読んで刺激を入れる**という方法でした。試験対策だけに閉じると飽きが早いので、「クラウドそのものが面白い」という感覚を意識的に維持する工夫が要ります。自分の興味が先にあって、試験は後からついてくる、という順番を崩さないことが、連投の最後まで走り切るコツだったように思います。

## HW寄りの立場で感じた「世界観の違い」

ここが記事の主題のひとつです。4資格を通して、デジタル設計側の視点とクラウドの考え方には、構造的な違いがあると感じました。

### レイヤーの跳躍感

まず単純に、触れているレイヤーの違いに驚きがあります。普段の業務は物理層に近いデジタル回路ですが、クラウドはその上にOS、仮想化、ネットワーク、マネージドサービスが積み重なっています。AZ-900で最初にクラウドに触れたときは、自分のレイヤーの上に積み重なる層のひとつひとつが抽象的で、全体像がつかみにくいというのが素直な印象でした。

特に印象的だったのは、**環境構築の手軽さ**です。デジタル回路として機能を実装するには、論理合成・タイミング収束・物理配置・検証……とリードタイムが相応にかかります。一方クラウドでは、数クリックでVMが立ち上がり、数分で冗長構成までできあがる。どちらが優れているという話ではなく、**時間スケールの前提がまったく違う世界**で仕事をしているのだと実感しました。

### 「使い切り前提」から「スケール前提」への頭の切り替え

HW設計は基本的にリソースを**使い切る前提**で組み立てます。チップ上の面積、消費電力、ピン数、メモリ容量はいずれも有限で、足りなければ機能を削るか次の世代まで待つか、という選択になります。

一方、クラウドは**スケール前提**で考えます。オートスケーリング、ロードバランサ、サーバーレスは、どれも「必要なときに必要な量を確保する」発想で成り立っています。AZ-104でオートスケーリングやLBを触ったとき、SAAでサーバーレスの設計原則を学んだときに、このメンタリティの違いを強く感じました。思考の向きが逆、と言ってもよい気がします。

### セキュリティの考え方の違い

業務でセキュリティ関連のモジュールに触れることもあり、クラウド資格の学習中にIAMやKMSといった機構に触れたとき、自然とHW側のセキュリティの考え方と比較する機会がありました。

結論として、**鍵の管理や暗号化など表層で似通う部分はある**ものの、クラウドとHWのセキュリティは**基本的に別の思想**で設計されています。AWSやAzureのセキュリティは**権限管理**が主軸です。誰が何にアクセスできるかをポリシーで定義し、違反を検知する方向で強化されます。一方HWのセキュリティは、**ハードウェア的な保護機構**が中心です。秘密の数値をチップの外に出さない、改ざんを物理的に成立させない、といった発想で構成されます。

興味深かったのは、クラウド側のネットワーク関連を掘り下げて学んでいた期間に、**日々の業務でも設計上の着想に波及があった**ことです。学習と業務が循環する体験は、この2ヶ月のなかでもとくに手応えがありました。

## これから受ける人へ

4資格を走り切って振り返ると、次のような感触があります。

- 「2ヶ月で4資格」はペース勝負というよりも、**先にスケジュールを決めて追いかけるゲーム**でした。速さそのものが目的ではなく、試験日確定によって学習が日々の予定に組み込まれる、という構造が本質だと思います
- ひとつずつじっくり取り組むやり方にも価値があります。自分はスピードを選びましたが、同じ4資格でも半年かけて取るのと2ヶ月で取るのでは、得られる手触りは違うはずです
- HW寄りのキャリアだから不利、という感覚は特にありませんでした。むしろ、ネットワークやOSなど業務で意識しきれていない領域を体系化できるため、向いている面すらあると思います
- 自分自身、受験を決めた時期に他の方の学習記録記事を読んで背中を押された経験があります。この記事もそういう位置づけで読んでもらえると嬉しいです
